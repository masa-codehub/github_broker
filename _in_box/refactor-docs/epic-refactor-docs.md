## As-is (現状)
- `github_broker`と`issue_creator_kit`のソースコードやドキュメントが同一リポジトリに混在しており、プロジェクトの全体像が把握しにくい状態になっている。
- 設計情報がADRやDesign Docに分散しており、各コンポーネントの仕様を網羅的に理解することが困難である。
- `plans/`フォルダが存在するが、現在の運用フローと乖離しており、メンテナンスされていない。

## To-be (あるべき姿)
- `github_broker`と`issue_creator_kit`の関心事が明確に分離され、それぞれのドキュメントが整理されたディレクトリ構造で管理されている。
- 各コンポーネント（`github_broker`, `issue_creator_kit`）について、第三者が見ても機能や実装を客観的に理解できる、網羅的で詳細な設計ドキュメントが存在する。
- プロジェクトの全体像と各コンポーネントへの導線がREADMEに明記されており、新規参画者が迅速にキャッチアップできる。

## ユーザーの意図と背景の明確化
ユーザーは、現状のままでは`github_broker`と`issue_creator_kit`の境界が曖昧で、今後の開発やメンテナンスにおいて混乱や手戻りが発生することを懸念している。
このリファクタリングを通じて、各コンポーネントの責務を明確にし、ドキュメントを体系的に整備することで、開発プロセス全体の透明性と効率性を向上させることを目的としている。

## 目標達成までの手順 (Steps to Achieve Goal)
1.  **Story: 既存ドキュメントの整理・移動:** 現在のADR/Design Docを精査・分類し、新しいディレクトリ構造に再配置する。
2.  **Story: 新規設計ドキュメントの作成:** 各コンポーネントの詳細な設計ドキュメントを新規に作成する。
3.  **Story: プロジェクト全体のドキュメント整備:** プロジェクトの入り口となるREADMEを更新し、全体の構成を分かりやすく示す。

## 完了条件 (Acceptance Criteria)
- このEpicを構成する全てのStoryの実装が完了していること。
- 最終的に、プロジェクトのドキュメントが、ユーザーの意図通りに整理され、誰にとっても分かりやすい状態になっていること。

## 成果物 (Deliverables)
- 整理されたドキュメントディレクトリ構造
- `github_broker`の詳細設計ドキュメント群
- `issue_creator_kit`の詳細設計ドキュメント群
- 更新された `README.md`
